# CommandEngine

**Модульный сборщик конфигураций для [Rasa](https://rasa.com/).**

[![Python](https://img.shields.io/badge/python-3.10%2B-blue.svg)](https://www.python.org/)
[![Rasa](https://img.shields.io/badge/target-Rasa%203.1-5A17EE.svg)](https://rasa.com/)
[![Status](https://img.shields.io/badge/version-1.0-green.svg)](#)

</div>

---

По умолчанию предлагается редактировать конфигурационные файлы `nlu.yml`, `domain.yml`, `rules.yml` и `stories.yml` внутри Rasa, которые очень сложно поддерживать при наличии большого количества намерений, сущностей и т.д. CommandEngine берёт на себя эту рутину: он собирает эти файлы из набора небольших, самодостаточных модулей. Каждый модуль описывает одну область поведения ассистента, например, погоду, музыку, приветствия в одном компактном файле `manifest.json`. Сборщик сканирует эти модули, проверяет их на согласованность и собирает из них полностью валидный набор конфигурационных файлов Rasa.

Логика остаётся рядом с данными, интенты не «размазаны» по трём разным YAML, а добавление новой команды - это создание одной папки, а не правка пяти файлов в разных концах проекта.

## Зачем это нужно

Rasa - мощный фреймворк, но по мере роста ассистента его конфигурация становится не поддерживаемой. Чтобы добавить одну команду «включи музыку», нужно вписать интент в `nlu.yml`, ответы и слоты в `domain.yml`, правило в `rules.yml`, а иногда ещё и историю в `stories.yml`. Данные одной функциональности физически лежат в четырёх файлах, и удержать их в согласованном состоянии становится всё труднее.

CommandEngine позволяет переосмыслить эту модель. Единица описания - модуль. Ниже приведена сравнительная таблица:

| Чистый Rasa | CommandEngine |
| --- | --- |
| Интенты - в `nlu.yml` | Всё про модуль «музыка» - в `modules/music/manifest.json` |
| Ответы и слоты - в `domain.yml` | Всё про модуль «погода» - в `modules/weather/manifest.json` |
| Правила - в `rules.yml` | Логика обработки - рядом, в `modules/*/handler.py` |
| Истории - в `stories.yml` | YAML для Rasa генерируется автоматически |

Вы редактируете читаемый JSON, разбитый по смыслу, а CommandEngine YAML-файлы собирает за вас - воспроизводимо и с проверками.

## Как это устроено

Сборка - это конвейер из четырёх последовательных стадий. Данные текут в одну сторону: из модулей в готовый проект Rasa.

1. **Scan** - обход `modules/`, чтение каждого `manifest.json` и общих данных из `_shared/`.
2. **Validate** - проверка манифестов по JSON-схеме и набор кросс-проверок.
3. **Aggregate** - слияние всех модулей в единое промежуточное представление (IR) - структуру со всеми интентами, сущностями, слотами, ответами, формами, правилами и историями.
4. **Serialize** - рендеринг IR в YAML-файлы.

Отдельным файлом является `registry.py` - он выполняет роль реестра всех возможных обработчиков, описанных в конфигурациях модулей. Этот файл используется во время диалога Rasa: внутри действия `action_dispatch` по имени интента определяется нужный обработчик в реестре и происходит его выполнение.

## Установка

Проекту нужны два независимых виртуальных окружения: CommandEngine (сборщик и панель) и сам Rasa предъявляют разные, конфликтующие требования к версии Python.

### 1. Основное окружение (CommandEngine)

Отсюда работают все папки проекта, кроме `rasa/`. Необходимо ставить пакет из `CommandEngine/`:

```bash
# cd CommandEngine/
python -m venv .venv
source .venv/bin/activate  # Linux
# .venv\Scripts\activate   # Windows

pip install -e .
```

Возможна установка через `uv` (в репозитории имеется `uv.lock`)

```bash
uv sync
uv pip install -e .
```

После установки в этом окружении появляются две консольные команды - `commandengine-build` и `commandengine-panel` (см. [Способы запуска](#способы-запуска)).

**Зависимости** (ставятся автоматически): `pydantic-settings`, `jsonschema`, `pyyaml`, `requests`, `fastapi`, `uvicorn[standard]`.

### 2. Окружение для Rasa

Отдельное окружение нужно, только чтобы взаимодействовать с самой Rasa - обучать модель на сгенерированных файлах и запускать ассистента. Создаётся оно внутри папки `rasa/` и использует Python не выше 3.10 (требование Rasa 3.x):

```bash
# cd CommandEngine/rasa/
py -3.10 -m venv .venv
source .venv/bin/activate  # Linux
# .venv\Scripts\activate   # Windows

pip install rasa

rasa init  
```

Инициализацию проекта Rasa (если папка rasa пуста) необходимо осуществлять перед использованием сборщика.

### Совместное использование обоих окружений

Сборку (`commandengine-build`) запускается в основном окружении - она создает `nlu.yml`, `domain.yml`, `rules.yml`, `stories.yml` в папке `rasa/`. Затем переключаетесь в окружение Rasa и уже там выполняете `rasa train`, `rasa run actions`, `rasa shell`.

## Быстрый старт

```bash
# Сборка конфигов Rasa из модулей
commandengine-build
# ИЛИ
# commandengine-panel # Панель на http://127.0.0.1:8000

# Обучение модели Rasa на сгенерированных файлах
rasa train

# Поднятие сервера кастомных действий
rasa run actions

# Запуск в режиме CLI-ассистента
rasa shell
# ИЛИ
# rasa run --enable-api # Для доступа через api
```

## Структура проекта

```
CommandEngine/
├── builder/
│   ├── config.py
│   ├── scanner.py
│   ├── validator.py
│   ├── aggregator.py
│   ├── serializator.py
│   ├── utils.py
│   ├── main.py
│   └── schemas/
│       └── manifest.schema.json
│
├── modules/
│   ├── _shared/
│   │   ├── responses.json
│   │   └── rules.json
│   ├── greeting/
│   │   └── manifest.json
│   ├── weather/
│   │   ├── manifest.json
│   │   └── handler.py
│   └── music/
│       ├── manifest.json
│       └── handler.py
│
├── panel/
│   ├── app.py
│   ├── run.py
│   ├── config.py
│   ├── routers/
│   ├── services/
│   └── static/
│
├── registry.py
│
├── rasa/
│   ├── data/
│   ├── domain.yml
│   ├── config.yml
│   ├── endpoints.yml
│   └── actions/actions.py
│
├── tests/
│
└── pyproject.toml
```

- **`builder/`** - ядро: сборщик модулей в конфиги Rasa (стадии `scan -> validate -> aggregate -> serialize`, схема манифеста в `schemas/`).
- **`modules/`** - контент модулей: в каждом их них `manifest.json` и опциональный обработчик `handler.py`; `_shared/` - данные, общие для всех модулей.
- **`panel/`** - веб-панель управления на FastAPI: приложение (`app.py`), запуск (`run.py`), HTTP-эндпоинты (`routers/`), логика (`services/`), фронтенд (`static/`).
- **`registry.py`** - реестр обработчиков для `action_dispatch`.
- **`rasa/`** - проект Rasa: `data/` и `domain.yml` генерируются сборщиком, `config.yml`/`endpoints.yml` - редактируются разработчиком, `actions/actions.py` - обработчик действий.
- **`pyproject.toml`** - метаданные пакета и CLI-скрипты.

Редактирование происходит в `modules/`, а `builder/` превращает их в содержимое `rasa/`. Файлы внутри `rasa/data/` и `rasa/domain.yml` перезаписываются при каждой сборке.

## Способы запуска

Сборщик можно запустить двумя способами: из командной строки или через веб-панель. Оба вызывают один и тот же код в `builder/`, поэтому результат идентичен.

`commandengine-build` - Полный конвейер: scan, validate, aggregate, serialize. Генерирует файлы Rasa.
`commandengine-panel` - Поднимает веб-панель управления на `FastAPI`.

### Командная строка (CLI)

Пример использования:

```bash
commandengine-build
```

Если валидация не проходит, сборщик печатает понятное сообщение и завершается с кодом 1:

```
BUILDING ERROR: The module 'weather', intent 'weather_ask': entity 'city' in example not declared: 'погода в [москве](city)'
```

Тот же билдер можно вызвать и как модуль, без установленных скриптов:

```bash
python -m builder.main
```

### Панель управления

Панель - это небольшое веб-приложение на FastAPI, через которое можно получить графический доступ к сборщику и просмотр модулей.

```bash
commandengine-panel  # Доступен по адресу http://127.0.0.1:8000
```

Возможности панели:

- Три кнопки-действия - «Сканировать», «Валидация», «Собрать» - вызывают соответствующие стадии конвейера и показывают лог выполнения прямо в браузере.
- Обозреватель модулей - список всех модулей с описанием состояния («без манифеста», «битый JSON»), а рядом - просмотр их файлов с подсветкой синтаксиса.

В будущем будет добавлен ещё функционал, позволяющий создавать новые модули и редактировать конфигурационные файлы прямо в браузере.

## Анатомия модуля

Модуль - это папка внутри `modules/`. Описание минимального модуля приведён ниже:

```
modules/
└── my_module/
    ├── manifest.json
    └── handler.py
```

В файле `manifest.json` приводится описание команд модуля, и он является обязательным. Также в модуле может находится файл-обработчик `handler.py` и он будет обязательным в случае упоминания обработчика в виде функции (`"type": "function"`) в манифесте.

## Файл манифеста

Манифест является важнейшим файлом модуля. Это обычный JSON-объект, который декларативно описывает, какие команды понимает модуль и что на них отвечать. Структура манифеста строго проверяется по JSON-схеме, лишние поля запрещены, это защищает от опечаток.

### Основная структура

На верхнем уровне манифест всегда содержит поле `module`. Всё остальное - это описание команд. Ниже приведён скелет самого простого манифеста:

```json
{
  "module": "greeting",
  "version": "1.0",

  "intent": "hello",
  "examples": [
    "привет", "здравствуй", "добрый день"
  ],
  "response": [
    "Привет!", "Здравствуйте!"
  ]
}
```

 - **`module`** - уникальное имя модуля. Должно совпадать с именем папки.
 - **`version`** - свободная версия модуля. На сборку не влияет, служит для учёта.
 - **`commands`** - список команд.
 - **`intent`, `examples`, `entities`, `slots`, `form`, `response`, `handler`, `stories`** - поля команд, лежат в массиве `commands`. Возможно определение этих полей на уровне поля `module`, если подразумевается одна команда на весь модуль (см. пример выше). В противном случае, поле `module` становится общим для всех команд модуля:

```json
{
  "module": "music",
  "entities": {
    "artist": { 
      "lookup": [
        "металлику", "queen", "кино"
      ]
    }
  },
  "commands": [
    {
      "intent": "play",
      "examples": [
        "включи музыку", "включи [queen](artist)"
      ],
      "handler": {
        "type": "function", 
        "name": "run_play"
      }
    },
    {
      "intent": "next",
      "examples": [
        "следующий трек", "дальше"
      ],
      "response": [
        "Включаю следующий трек."
      ]
    }
  ]
}
```

Здесь сущность `artist` объявлена один раз наверху в общем `entities`, она доступна в любой команде (глобальное определение сущности). Если команда объявит свою сущность с тем же именем (локальное опредение сущности), её определение переопределит общее.

### Три типа команд

Тип команды не указывается явно, сборщик выводит его из набора полей в манифесте. Это определяет, какие правила и ответы будут сгенерированы для Rasa:

| Тип | Набор полей | Что происходит при диалоге |
| --- | --- | --- |
| статическая | есть `response`, нет `handler` | Rasa сразу отвечает готовым текстом из `response`. Python-код не выполняется. |
| динамическая | есть `handler`, нет `form` | Rasa вызывает `action_dispatch`, тот запускает функцию из `handler.py`. Ответ формируется кодом. |
| форма | есть `form` и `handler` | Rasa сначала собирает недостающие слоты через диалог (форма), затем вызывается обработчик. |

> Команда обязана иметь хотя бы одно из полей `handler`, `form` или `response`, иначе будет ошибка сборки.

### Справочник по полям

Ниже приведена таблица всех полей команды с их описанием:

| Поле | Тип | Описание |
| --- | --- | --- |
| `intent` | строка | Имя интента внутри модуля. В финальном конфиге Rasa станет `<module>_<intent>`. |
| `examples` | массив строк | Обучающие фразы для NLU. |
| `entities` | объект | Сущности, которые встречаются в примерах (см. ниже). |
| `slots` | объект | Память диалога, куда попадают распознанные значения (см. ниже). |
| `form` | объект | Описание формы для пошагового сбора слотов (см. ниже). |
| `response` | массив строк | Готовый текст ответа для статической команды. |
| `handler` | объект | Привязка к Python-функции или HTTP-эндпоинту (см. ниже). |
| `stories` | массив объектов | Истории Rasa, если нужен сложный многошаговый сценарий. |

#### `entities`

Сущность - это фрагмент фразы, который нужно извлечь: город, исполнитель, число. Разметка в примерах - `[текст](имя)`:

```json
"entities": {
  "city": {
    "lookup": [
      "москва", "санкт-петербург", "казань", "новосибирск"
    ],
    "synonyms": {
      "санкт-петербург": [
        "питер", "спб"
      ]
    }
  }
}
```

 - **`lookup`** - список известных значений. Rasa использует его как подсказку - сущность лучше распознаётся, даже вне обучающих примеров.
 - **`synonyms`** - отображение `каноническое_значение -> варианты_написания`. В примере «питер» и «спб» будут нормализованы в «санкт-петербург».

Каждая сущность, размеченная `[..](name)` в примерах, обязана быть объявлена - либо здесь, либо в общих сущностях `_shared`.

#### `slots`

Слот - это именованная ячейка состояния диалога. Из слотов обработчик в коде на Python читает значения, а формы - собирают недостающие данные.

```json
"slots": {
  "city": {
    "type": "text",
    "from": "entity"
  }
}
```

| Ключ | Значения | Назначение |
| --- | --- | --- |
| `from` | `entity` / `text` / `intent` | Место, откуда заполнять слот: из распознанной сущности, из всего текста или из факта срабатывания интента. |
| `type` | `text` / `bool` / `float` / `categorical` / `list` / `any` | Тип слота в Rasa. По умолчанию `text`. |
| `entity` | строка | Имя сущности-источника (если отличается от имени слота). Работает с `from: entity`. |
| `values` | массив строк | Допустимые значения для `type: categorical`. |

#### `form`

Форма нужна, когда для выполнения команды не хватает данных и их надо доспросить у пользователя. Классический пример: заказ такси: нужны и адрес отправления, и адрес назначения.

```json
"slots": {
  "from_city": { 
    "type": "text", 
    "from": "entity", 
    "entity": "city" 
  },
  "to_city": { 
    "type": "text", 
    "from": "entity", 
    "entity": "city" 
  }
},

"form": {
  "required": [
    "from_city", "to_city"
  ],
  "ask": {
    "from_city": "Откуда едем?",
    "to_city": "Куда едем?"
  }
},

"handler": { 
  "type": "function", 
  "name": "run" 
}
```

| Ключ | Тип | Назначение |
| --- | --- | --- |
| `required` | массив строк | Слоты, которые форма обязана заполнить перед завершением. |
| `ask` | объект | Вопрос, который бот задаёт, чтобы получить каждый слот. Ключ - имя слота, значение - текст вопроса. |
| `validate` | объект | Зарезервировано под кастомную валидацию значений слотов. |

Когда пользователь вызывает интент формы, Rasa активирует цикл сбора слотов, по очереди задавая вопросы из `ask`, а после заполнения всех `required` - передаёт управление обработчику.

#### `response`

Самый простой способ ответить, без единой строчки Python. Если строк несколько, Rasa выбирает случайную:

```json
"response": [
  "Привет!", "Здравствуйте!", "Рад тебя видеть!"
]
```

#### `handler`

Связывает команду с логикой. Есть два типа:

 - Вариант 1: локальная Python-функция из `handler.py` этого модуля
```json
"handler": { 
  "type": "function", 
  "name": "run" 
}
```

 - Вариант 2: внешний HTTP-сервис
```json
"handler": { 
  "type": "http", 
  "url": "https://api.example.com/weather" 
}
```

| Ключ | Назначение |
| --- | --- |
| `type` | `function` - вызов функции из `handler.py`; `http` - POST-запрос на внешний URL. |
| `name` | Имя функции в `handler.py`. Если не указано, берётся `run`. |
| `url` | Адрес эндпоинта. CommandEngine отправит туда POST с телом `{"slots": {...}, "entities": {...}}` и вернёт JSON-ответ пользователю. |

## Обработчики `handler.py`

`handler.py` - это обычный Python-модуль, находящийся рядом с манифестом. В нём пишутся функции, которые вызываются, когда сработала динамическая команда. Важно соблюдать сигнатуру и формат возвращаемого значения.

**Обязательная сигнатура:**

```python
def run(slots: dict, entities: dict) -> dict:
    ...
    return {"text": "ответ пользователю"}
```

- Принимает два аргумента: `slots` - заполненные слоты диалога и `entities` - сущности, распознанные в последней реплике.
- Возвращает `dict`. Ключ `text` - то, что бот скажет пользователю. Опциональный ключ `events` - список событий Rasa.

> При обращении к `slots` или `entities` рекомендуется использовать `.get()` с запасным значением: слот или сущность могут быть не заполнены.

## Общие данные: каталог `_shared`

Некоторые сущности, правила, истории и т.д. не принадлежат ни одному конкретному модулю: к примеру, глобальный fallback на непонятную фразу, общий словарь городов. Для них есть `modules/_shared/` - эта папка не считается модулем.

Все четыре файла опциональны. Ниже приведено описание этих файлов:

 - **`responses.json`** - общие ответы Rasa (`utter_*`).
 - **`rules.json`** - общие правила Rasa.
 - **`entities.json`** - общие сущности, доступные всем модулям, пишутся в том формате, что `entities` в манифесте.
 - **`stories.json`** - общие истории Rasa.

**Пример `_shared/responses.json`** - ответ по умолчанию, когда бот не понял реплику:

```json
{
  "utter_default": [
    { 
      "text": "Извините, не понял команду. Повторите, пожалуйста." 
    }
  ]
}
```

**Пример `_shared/rules.json`** - правило, связывающее срабатывание fallback c ответом по умолчанию:

```json
[
  {
    "rule": "fallback на непонятную фразу",
    "steps": [
      { 
        "intent": "nlu_fallback" 
      },
      { 
        "action": "utter_default"
      }
    ]
  }
]
```

> Общие сущности (`entities.json`) имеют полезный эффект: если несколько модулей используют сущность, объявите её один раз здесь, и все модули смогут ссылаться на `[..](entity)` в примерах, не дублируя lookup-таблицу.

## Результат работы сборщик

После `commandengine-build` при успехе в каталоге `rasa/` появляются следующие файлы:

 - **`rasa/data/nlu.yml`** - нтенты и их примеры и lookup-таблицы и синонимы из сущностей.
 - **`rasa/domain.yml`** - интенты, сущности, слоты (с mapping), ответы, формы, действие `action_dispatch`, session config.
 - **`rasa/data/rules.yml`** - правила: по одному на каждую команду (static/dynamic/form) и общие правила из `_shared`.
 - **`rasa/data/stories.yml`** - истории из команд и `_shared`. Если историй нет - файл удаляется, чтобы не мусорить.

Генерируемое по типам команд:

- **static** - ответ `utter_<module>_<intent>` в `domain.yml` и правило `интент -> произнести_ответ`.
- **dynamic** - правило `интент -> action_dispatch`.
- **form** - форма с `required_slots` в `domain.yml`, вопросы для заполения слотов в виде `utter_ask_<slot>` из блока `ask` и два правила: активация формы и её завершение с последующим вызовом `action_dispatch`.

## Правила валидации

Стадия `validate` ловит ошибки до генерации файлов Rasa и обучения модели. Проверки делятся на две группы.

**Структурные (по JSON-схеме) - `validate_each`:**

- Манифест соответствует `manifest.schema.json`: нет посторонних полей, типы верны, имена интентов и модулей подчиняются шаблону.
- Имя модуля в манифесте совпадает с именем папки.
- Команда не смешивает стили: нельзя одновременно иметь `commands` и поля команды верхнего уровня.
- У каждой команды есть `intent` и `examples`.

**Смысловые (кросс-проверки) - `check_collisions`:**

- Каждая сущность, размеченная `[..](name)` в примерах, объявлена в модуле или в `_shared`.
- У формы объявлены слоты; все `required`-слоты присутствуют в `slots`; все слоты из `ask` входят в `required`.
- Если `handler.type` равен `function`, рядом с манифестом есть `handler.py`.

Любая проваленная проверка поднимает `BuildError` с точным адресом проблемы - модуль, интент и суть ошибки.

## Внутренняя архитектура сборщика

Для тех, кто хочет расширять CommandEngine, - краткая карта модулей `builder/` и их ответственности. Каждый файл соответствует одной стадии конвейера, что упрощает навигацию и тестирование.

| Модуль | Ключевые элементы | Пояснение |
| --- | --- | --- |
| [`config.py`](builder/config.py) | `Settings` | Настройки, необходимые сборщику: имена, версии, все пути в файловой системе, регулярное выражение разметки сущностей. Построено на основе `pydantic-settings`. |
| [`scanner.py`](builder/scanner.py) | `scan`, `_load_shared` | Читает манифесты модулей и данные `_shared`. Возвращает `(manifests, shared)`. Помечает каждый манифест служебным ключом `__dir` - абсолютный путь к папке. |
| [`validator.py`](builder/validator.py) | `validate`, `validate_each`, `check_collisions` | Все проверки корректности (см. выше). |
| [`aggregator.py`](builder/aggregator.py) | `aggregate_in_ir` | Сливает модули в IR - словарь со списками интентов, сущностей, слотов, ответов, форм, правил, историй. На этом этапе вычисляются итоговые имена для конфигов Rasa (`<module>_<intent>`, `<...>_form`). |

| [`main.py`](builder/main.py) | `build`, `main` | Выполняет конвейер в правильном порядке и служит точкой входа CLI. |

## HTTP API панели

Панель создаёт небольшой REST API. Все ответы - JSON.

**Действия сборщика** ([`actions_router.py`](panel/routers/actions_router.py)):

| Метод | Путь | Действие |
| --- | --- | --- |
| `POST` | `/api/scan` | Сканировать модули, вернуть сводку по количеству. |
| `POST` | `/api/validate` | Сканировать и валидировать без генерации файлов. |
| `POST` | `/api/build` | Полная сборка. |

Формат ответа: `{ "ok": bool, "log": "...", "data"?: {...}, "error"?: "..." }`. При ошибке - статус `400` и текст в `error`. Знак ? обозначает необязательность поля.

**Обозреватель модулей** ([`modules_router.py`](panel/routers/modules_router.py)):

| Метод | Путь | Действие |
| --- | --- | --- |
| `GET` | `/api/modules` | Список модулей с метаданными (есть ли манифест/handler, валиден ли JSON, список интентов). |
| `GET` | `/api/modules/{module}` | Детали модуля: файлы + разобранный манифест. |
| `GET` | `/api/modules/{module}/files/{file}` | Содержимое файла модуля (для просмотра). |
| `PUT` | `/api/modules/{module}/files/{file}` | Сохранение файла - пока не реализовано (всегда возвращает `501`). |

Чтение файлов защищено от path traversal: имена модулей и файлов проверяются, а расширения ограничены белым списком (`.json`, `.py`, `.md`, `.yml`, `.yaml`, `.txt`).
