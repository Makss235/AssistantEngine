## Предпосылки

Rasa запускается в отдельном виртуальном окружении внутри этой папки (`rasa/`), потому что требуется Python не выше 3.10 - это конфликтует с основным окружением CommandEngine. Настройка окружения описана в [README CommandEngine, раздел «Установка»](../README.md#установка). Кратко:

```bash
# cd CommandEngine/rasa/
py -3.10 -m venv .venv
source .venv/bin/activate  # Linux
# .venv\Scripts\activate   # Windows
pip install rasa
```

## Полный цикл: от сборки до диалога

Сборка выполняется в основном окружении, обучение и запуск - в окружении Rasa. Обычно это два-три параллельных терминала.

```bash
# [окружение CommandEngine]  Собрать конфигурационные файлы из модулей
commandengine-build

# [окружение Rasa]  Обучить модель на сгенерированных файлах
rasa train

# [окружение Rasa, отдельный терминал]  Поднять сервер действий
rasa run actions

# [окружение Rasa, ещё один терминал]  Запустить ассистент в CLI-режиме
rasa shell
```

Каждый раз, когда меняются модули и происходит пересборка с помощью команды (`commandengine-build`) или из панели, модель нужно переобучить (`rasa train`) - иначе Rasa продолжит работать на старой модели из `models/`.

## Что генерируется, а что редактируется руками

| Файл / папка | Кто им управляет | Комментарий |
| --- | --- | --- |
| `data/nlu.yml` | **Сборщик** | Интенты, примеры, lookup-таблицы, синонимы. |
| `data/rules.yml` | **Сборщик** | По правилу на команду; правила из `_shared`. |
| `data/stories.yml` | **Сборщик** | Создаётся, только если есть истории; иначе не создаётся/удаляется. |
| `domain.yml` | **Сборщик** | Интенты, сущности, слоты, ответы, формы, `action_dispatch`. |
| `config.yml` | **Разработчик** | Пайплайн NLU и policies. |
| `endpoints.yml` | **Разработчик** | Адрес сервера действий, tracker store и т.п. |
| `credentials.yml` | **Разработчик** | Каналы связи (rest, telegram, …). |
| `actions/actions.py` | **Разработчик** | Единая точка входа `action_dispatch`. Обычно не трогается. |
| `models/` | **Rasa** (`rasa train`) | Обученные модели, артефакты. |

## Конфигурационные файлы

### `config.yml`

Задаёт пайплайн NLU и policies. В текущей конфигурации:

- `language: ru` - язык ассистента.
- `recipe: default.v1`, `pipeline: null`, `policies: null` - используются пайплайн и policies Rasa по умолчанию.
- `assistant_id` - уникальный идентификатор ассистента, генерируется автоматически.

По умолчанию, пайплайн включает `FallbackClassifier` с порогом `0.3`. Именно он выдаёт интент `nlu_fallback`, на который в `data/rules.yml` завязано правило из `_shared`: ответ `utter_default`.

### `endpoints.yml`

Точки подключения внешних сервисов. Активна одна:

```yaml
action_endpoint:
  url: "http://localhost:5055/webhook"
```

Это адрес сервера кастомных действий (`rasa run actions` по умолчанию слушает порт `5055`). Блоки `tracker_store` и `event_broker` закомментированы - значит, история диалога хранится в памяти и теряется при перезапуске. Для сохранения состояния подключите Redis/Mongo (заготовки есть в файле).

### `credentials.yml`

Каналы, через которые с ботом можно общаться. Включён канал `rest`:

```yaml
rest:
```

Он открывает HTTP-эндпоинт `POST /webhooks/rest/webhook` (см. [HTTP API](#http-api-rasa-run---enable-api)). Заготовки под Facebook, Slack, Mattermost и Rasa Enterprise присутствуют закомментированными.

## Обучение модели: `rasa train`

```bash
rasa train
```

Команда читает `config.yml`, `domain.yml` и всё из `data/`, обучает NLU- и Core-модель и складывает архив в `models/` (имя - по времени обучения). Ассистент при запуске берёт последнюю модель из `models/`.

Полезные варианты:

```bash
rasa train nlu      # обучить только NLU
rasa train --force  # переобучить всё, игнорируя кэш
rasa data validate  # проверить согласованность data/ и domain.yml без обучения
```

## Сервер действий: `rasa run actions`

Динамические команды (`handler`) и формы выполняются не самой Rasa, а отдельным процессом - сервером действий `rasa_sdk`:

```bash
rasa run actions  # слушает http://localhost:5055/webhook
```

Он загружает [`actions/actions.py`](actions/actions.py), где объявлено единственное действие `action_dispatch`. Держите этот процесс запущенным всё время, пока работает ассистент, - иначе на любой динамической команде Rasa вернёт ошибку обращения к серверу действий.

Статические команды сервер действий не используют - Rasa отвечает готовым текстом сама.

## Как работает `action_dispatch`

Вместо отдельного `Action`-класса на каждую команду CommandEngine генерирует правила вида `интент -> action_dispatch`, а сам диспетчер по имени интента находит нужный обработчик в реестре. Логика ([`actions/actions.py`](actions/actions.py)):

1. Из трекера берётся имя сработавшего интента (`<module>_<intent>`), заполненные слоты и сущности последней реплики.
2. В `REGISTRY` по имени интента ищется функция-обработчик.
3. Обработчик вызывается как `handler(slots, entities)` и возвращает `dict`; ключ `text` отправляется пользователю, опциональный `events` - в трекер Rasa.
4. Если обработчика нет - бот отвечает «Команда пока не поддерживается.».

`REGISTRY` собирается из манифестов всех модулей: для `handler.type = function` берётся функция из `handler.py` модуля, для `handler.type = http` создаётся обёртка, шлющая `POST {"slots": ..., "entities": ...}` на указанный URL. Подробнее об обработчиках - в [README CommandEngine](../README.md#обработчики-handlerpy).

## Запуск ассистента

### CLI: `rasa shell`

Диалог в терминале - самый быстрый способ проверить бота:

```bash
rasa run actions  # терминал 1
rasa shell        # терминал 2
```

`rasa shell` использует последнюю модель из `models/`. Для динамических команд обязателен запущенный сервер действий.

### HTTP API: `rasa run --enable-api`

Поднимает ассистента как HTTP-сервис (по умолчанию порт `5005`):

```bash
rasa run --enable-api  # требуется отдельно rasa run actions
```

Отправка сообщения через включённый канал `rest`:

```bash
curl -X POST http://localhost:5005/webhooks/rest/webhook \
  -H "Content-Type: application/json" \
  -d '{"sender": "user1", "message": "погода в москве"}'
```

В ответ придёт JSON-массив реплик бота, например:

```json
[{"recipient_id": "user1", "text": "В городе москва сейчас +5°C, без осадков."}]
```
